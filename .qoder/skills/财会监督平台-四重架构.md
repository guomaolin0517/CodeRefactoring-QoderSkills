## 1. 项目概述

本项目是一个基于 **Spring Cloud** 的微服务架构系统，主要用于**财会监督平台**的管理和运营。本文档定义了将其重构为**标准四重架构**（底座层、能力层、聚合层、体验层）的完整规划方案。

**核心定位**：一个核心、四大维度、六大模块、一个闭环

- **一个核心**：以"财会监督闭环管理"为核心
- **四大维度**：基础支撑、监督业务、辅助保障、智能分析
- **六大模块**：规则库、问题库、考评库、风险防控、数据支撑、档案管理
- **一个闭环**：配置 -> 执行 -> 处置 -> 考评 -> 归档 -> 优化

## 3. 四重架构定义

### 3.1 四层架构概念

| 层级 | 目录前缀 | 作用域 | 职责 |
|------|----------|--------|------|
| **底座层** (Foundation) | `grp-common-boot/` / `grp-common-{module}/` | 全局/模块级 | 通用基础设施（日志、工具、数据库、Redis、AI等）+ 模块级通用底座 |
| **能力层** (Capability) | `grp-capability-{module}/` | 模块级 | 原子业务能力（Model、Controller、Service） |
| **聚合层** (Aggregation) | `grp-aggregation-{module}/` | 模块级 | 服务启动编排（Spring Cloud 适配） |
| **体验层** (Experience) | `grp-experience-{module}/` | 模块级 | 对外 Feign SDK 封装 |

### 3.2 层级依赖方向

```
体验层 (Experience)
  └── {module}-feign-com  ──依赖──>  grp-{module}-model

聚合层 (Aggregation)
  └── {module}-server-springcloud  ──依赖──>  {module}-controller

能力层 (Capability)
  ├── {module}-controller  ──依赖──>  {module}-service
  ├── {module}-controller  ──依赖──>  grp-{module}-model
  ├── {module}-service     ──依赖──>  grp-{module}-model
  └── {module}-service     ──依赖──>  底座层模块

底座层 (Foundation)
  └── grp-common-boot 下各子模块（独立，无业务依赖）
```

### 3.3 核心原则

- **不改变业务逻辑**：只调整工程结构、目录组织、命名规范和代码质量
- **铁律三类文件**：只调整 `pom.xml`、`*.yml/*.yaml/*.properties`、`*.java` 的 import/package
- **自底向上**：底座层 -> 能力层 -> 聚合层 -> 体验层，按顺序构建

### 4.3 当前架构问题分析

| 问题 | 影响 |
|------|------|
| 扁平目录结构 | 模块职责不清晰，底座/能力/聚合/体验混杂 |
| ufsp-config 承载过多 | 配置服务依赖了过多业务模块，职责边界模糊 |
| ufsp-business 定位不明 | 业务入口与聚合层职责未分离 |
| 公共模块散落 | util/cache/constant/enums 等散落在各业务模块中 |
| Feign 统一模块 | ufsp-feign 作为单一模块，未按业务域拆分 |
| 缺少标准分层 | Controller/Service/DAO 层未严格分离 |

## 8. 9大技能重构执行计划

### 8.1 执行顺序总览

| 序号 | 技能名称 | 作用层级 | UFSP 具体执行内容 |
|------|----------|---------|-------------------|
| Step1 | 工程重构 | 工程级 | 将 UFSP 扁平结构重构为四层架构目标目录 |
| Step2 | 工程架构微调 | 工程级 | server->controller, server-com->service 模块重命名 |
| Step3 | 架构依赖守卫 | 代码级 | 检查修复 Controller->Controller 等分层依赖违规 |
| Step4 | DAO 层治理 | 代码级 | DAO 层目录治理 + model 实体集中到 model 模块 |
| Step5 | Service 层治理 | 代码级 | Service 层 facade/impl 接口实现分离 |
| Step6 | Controller 层治理 | 代码级 | Controller custom/common 接口分离 |
| Step7 | 接口与命名规范 | 代码级 | 接口路径、类/属性命名、Bean 冲突检查修复 |
| Step8 | 公共模块提取 | 代码级 | util/cache/constant/enums/exception/config 提取到 common |
| Step9 | 工程代码优化 | 代码级 | SQL注入修复、日志增强、冗余代码清理 |

### 8.2 流程依赖关系

```
Step 1: 工程重构（四层架构搭建）
    |   将 ufsp 扁平模块重组为 {module}-module 四层容器
    v
Step 2: 工程架构微调（模块重命名）
    |   server -> controller, server-com -> service
    v
Step 3: 架构依赖守卫（分层依赖修复）
    |   消除 Controller->Controller、Controller->DAO 等违规依赖
    v
Step 4: DAO 层治理（DAO 目录归位 + model 集中）
    |   imp->impl, mapper/entity 分离, model 实体集中到 model 模块
    v
Step 5: Service 层治理（接口/实现分离）
    |   Service 接口归入 facade/, 实现归入 impl/
    v
Step 6: Controller 层治理（接口分离）
    |   Controller 按 custom/common 分离
    v
Step 7: 接口与命名规范（命名修复）
    |   四级路径、大/小驼峰、Bean 冲突排查
    v
Step 8: 公共模块提取（公共代码归集）
    |   util/cache/constant/enums/exception/config 提取到 grp-common-{module}
    v
Step 9: 工程代码优化检查（代码质量优化）
        SQL 注入修复、@Slf4j 统一、冗余代码清理
```

### 8.3 各 Step 对 UFSP 的具体影响

#### Step 1: 工程重构

**输入**：当前 UFSP 扁平目录结构
**输出**：四层架构标准目录结构

**具体操作**：
1. 创建根 POM（含 dependencyManagement 统一版本管理）
2. 创建 `grp-common-boot/` 底座层，移入 ufsp-common、ufsp-framwork-*、ufsp-api、ufsp-magic
3. 创建 `baseinfo-module/`、`problem-module/`、`evaluation-module/` 等业务模块容器
4. 每个业务模块容器下创建 grp-common/grp-capability/grp-aggregation/grp-experience 四个子容器
5. 将现有叶子模块移入对应容器
6. 更新所有 POM 的 parent/modules/dependencies
7. 更新 Java 文件的 package 和 import（如有变更）
8. 编译验证

**聚合层处理**：
- `ufsp-config` -> 拆分为各模块的 `*-server-springcloud`（配置管理功能下沉到对应业务模块的 service 层）
- `ufsp-business` -> 拆分为各模块的 `*-server-springcloud`（业务编排功能下沉到对应 controller 层）

#### Step 2: 工程架构微调

**具体操作**：
- `{module}-server` -> `{module}-controller`
- `{module}-server-com` -> `{module}-service`
- 同步更新 POM 引用和 Java package/import

#### Step 3: 架构依赖守卫

**UFSP 重点检查项**：

| 检查项 | UFSP 可能存在的违规 |
|--------|-------------------|
| Controller->Controller 直接依赖 | 配置类 Controller 可能直接调用业务 Controller |
| Controller 直接依赖 DAO/Mapper | Controller 中可能直接注入 Mapper 操作数据库 |
| Controller 注入 ServiceImpl | 未面向接口编程 |
| Entity 泄露到 Controller 层 | Controller 方法直接返回数据库 Entity |

#### Step 4: DAO 层治理

**UFSP 重点操作**：
- 修正命名（如 `imp` -> `impl`）
- DAO 层 mapper/entity 分离到标准子目录
- 将各模块 model 实体集中到 `grp-{module}-model`（不改变 package）

#### Step 5: Service 层治理

**目标目录结构**：
```
{module}-service/
└── src/main/java/
    └── com/ufsp/{module}/service/
        ├── facade/           # Service 接口
        │   ├── IXxxService.java
        │   └── IYyyService.java
        └── impl/             # Service 实现
            ├── XxxServiceImpl.java
            └── YyyServiceImpl.java
```

#### Step 6: Controller 层治理

**UFSP Controller 分离方案**：
```
{module}-controller/
└── src/main/java/
    └── com/ufsp/{module}/controller/
        ├── custom/           # 外部接口（面向前端/第三方）
        │   ├── problem/      # 问题管理相关
        │   ├── evaluation/   # 考评相关
        │   └── {business}/   # 其他业务分组
        └── common/           # 内部接口（面向内部微服务）
            ├── api/          # 内部 API
            ├── util/         # 工具/调试
            └── {function}/   # 其他内部功能
```

#### Step 7: 接口与命名规范

**UFSP 重点规范项**：
- 接口路径四级规范：`/{module}/{version}/{resource}/{action}`
- 类命名后缀规范：Controller/Service/Mapper/DTO/VO
- 属性命名小驼峰
- Bean 命名冲突排查（多模块合并后可能产生冲突）

#### Step 8: 公共模块提取

**UFSP 提取范围**：

| 提取包 | 从哪些模块提取 | 目标模块 |
|--------|--------------|----------|
| `util/` | 各业务 Service 模块中的工具类 | `grp-common-{module}/` |
| `cache/` | 缓存相关代码 | `grp-common-{module}/` |
| `constant/` | 常量定义 | `grp-common-{module}/` |
| `enums/` | 枚举定义 | `grp-common-{module}/` |
| `exception/` | 异常定义 | `grp-common-{module}/` |
| `config/` | 配置类 | `grp-common-{module}/` |

#### Step 9: 工程代码优化

**UFSP 重点优化**：
- SQL 注入修复（MyBatis `${}` -> `#{}` 参数化）
- 统一 `@Slf4j` 注解
- StringBuffer -> StringBuilder
- 冗余变量消除

## 10. 总结

### 10.1 重构前后对比

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| 目录结构 | 扁平，20+ 模块并列 | 四层架构，按业务域组织 |
| 职责边界 | 模块职责交叉 | 底座/能力/聚合/体验分层清晰 |
| 依赖关系 | 随意依赖 | 严格分层依赖，禁止跨层/同层直接引用 |
| 跨域调用 | 直接类引用 | 通过 Feign 接口调用 |
| 公共代码 | 散落各模块 | 统一收归 grp-common-boot 和 grp-common-{module} |
| 版本管理 | 各模块自行声明版本 | 根 POM dependencyManagement 统一管理 |
| 代码规范 | 命名/路径不统一 | 经 Step3~Step9 治理后全面规范化 |
